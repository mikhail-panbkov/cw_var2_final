# Сценарий демонстрации на защите (вариант 2, индекс B+-tree)

Памятка: что и в каком порядке показывать, какие команды выполнять.

## 0. Сборка

```powershell
cmake -G Ninja -S . -B build
cmake --build build
```

Появятся три бинарника: `cwdb` (автономный), `cwdb-server`, `cwdb-client`.

## 1. Главный батч-сценарий (показывает почти всё)

```powershell
.\build\cwdb.exe examples\defense_demo.sql
```

Один прогон демонстрирует: создание БД/таблицы со всеми модификаторами
(`INDEXED`, `NOT_NULL`, `DEFAULT`), вставки (полная, частичная с DEFAULT и NULL,
многострочная), вывод JSON, поиск по B+-tree (`==`, `!=`, `<`, `<=`, `>`, `>=`,
`BETWEEN`), лексикографическое сравнение строк, `LIKE` (RegEx), составные
условия (`AND`/`OR`/скобки), `UPDATE`, `DELETE`, агрегаты (`SUM`/`COUNT`/`AVG`),
квалифицированное имя `db.table`, валидацию ошибок (без падения программы),
телеметрию (`SHOW TELEMETRY`) и `REVERT`.

Карта соответствия пунктам задания — в конце файла.

## 2. Файл индекса B+-tree (вариант 2 — обязательно показать)

После прогона откройте файл индекса — он обновляется по ходу работы БД:

```powershell
type data\demo\students\index_id.bpt
```

Видно структуру дерева: строку `ORDER`, внутренние узлы (`keys=[...]
children=[...]`) и листья (`{ключ=row}` со связью `next`). Поясните: ключ —
значение столбца, значение — номер строки (индекс не дублирует данные).

## 3. Журнал доступа (Access Logs)

```powershell
type data\access.log
```

Покажите поля по каждому запросу: время начала/конца, идентификаторы запроса и
транзакции, клиент, обработчик, код возврата, статус и тело запроса.

## 4. Темпоральная персистентность — содержательный REVERT (интерактивно)

REVERT восстанавливает таблицу из ближайшего снимка с меткой `<=` указанной.
Чтобы показать реальный откат, нужна настоящая метка. Самый наглядный способ:

```powershell
.\build\cwdb.exe
```
```sql
CREATE DATABASE t1;
USE t1;
CREATE TABLE p (id int INDEXED, v string);
INSERT INTO p (id, v) VALUE (1, "before");
SELECT * FROM p;          -- состояние "before"
```

Теперь откройте журнал снимков и скопируйте ПОСЛЕДНЮЮ метку (строка вида
`@2026.05.30-14:23:01.123`, берётся часть после `@`):

```powershell
type data\t1\_db_history.log
```

Вернитесь в сеанс, измените данные и откатитесь к скопированной метке:

```sql
UPDATE p SET v = "after" WHERE id == 1;
SELECT * FROM p;                              -- состояние "after"
REVERT p 2026.05.30-14:23:01.123;            -- ВСТАВЬТЕ свою метку
SELECT * FROM p;                              -- снова "before" — откат сработал
EXIT;
```

Подчеркните: восстановление логическое (из снимка содержимого в
`_db_history.log`), а НЕ копирование файлов БД — как требует задание.

## 5. Клиент-серверный режим (задание 3)

Терминал 1 — запустить сервер:
```powershell
.\build\cwdb-server.exe 127.0.0.1 5429
```

Терминал 2 — подключить клиент:
```powershell
.\build\cwdb-client.exe --host 127.0.0.1 --port 5429
```
```sql
CREATE DATABASE net; USE net;
CREATE TABLE t (id int INDEXED, name string NOT_NULL);
INSERT INTO t (id, name) VALUE (1, "Ann"), (2, "Bob");
SELECT * FROM t WHERE id >= 1;
SHOW TELEMETRY;
EXIT;
```

Покажите, что `EXIT` закрыл только клиентское соединение, а сервер продолжает
работать: подключите второго клиента и убедитесь, что база `net` на месте
(общее состояние сервера). Клиент умеет и пакетный режим:

```powershell
.\build\cwdb-client.exe --host 127.0.0.1 --port 5429 examples\defense_demo.sql
```

## 6. String Interning (задание 2)

Это внутренняя оптимизация памяти: одинаковые строки хранятся в единственном
экземпляре (`InternPool`), ячейки ссылаются на них. Демонстрируется на словах —
покажите класс `InternPool` в `src/engine.hpp` и поясните, что повторяющиеся
значения (например, `faculty "IU7"` у нескольких студентов) не дублируются в
памяти.
